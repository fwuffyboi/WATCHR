ADMIN CONSOLE

The admin console will look simplistic and small, after all, it's an admin console. You know what you're doing.

The theorised first design will (hopefully) look as drawn below. Btw this IS the first drawing. This drawing was drawn to a 9 (length) x 5 (height) inch ratio.

The [ID:PASSWORD] field is a text field, you should use this field everytime you would like to send a command to the camera otherwise, it will not complete the desired operation. Every user will be provided with an id and password which is completely random and has different permission options. These ID’s and passwords will be stored ONLY onboard the actual camera. After each use of a command, the text field will clear to prevent id-stealing / using unauthorized commands.

The [upload] button will encrypt (with a 4096 bit key) and send your credentials to the camera, which will decrypt and check if they are correct. if they are, your permissions will be checked too, if you are allowed to use that command, it will be applied and used. Else, the admin console should receive a message explicitly stating that the signed in user is not allowed to use that command, and that the request has been logged.

The [help] button shows a pop up of how to use all text fields. It will also show all commands you can use with your current user id.

The [ENTER COMMAND] is a text field, you may enter any of the following commands into it:

  • GAD OFF (Turns off gender and age detection.)
  • GAD ON (Turns on gender and age detection)
  • PRIV OFF (Turns off privacy mode)
  • PRIV ON (Turns on privacy mode)
  • SHOW FEED (Show camera’s live feed.)
  • GET LOGS (Download all text logs)
  • GET IMAGE LOGS (Download all captured images into a single folder. (MAKE SURE TO SHOW AN ALERT POPUP SAYING HOW MANY IMAGES AND THE FOLDER SIZE))
  • GET RESO (opens a popup showing all resources being used such as; CPU, RAM, camera runtime, and the time in that location of the camera, and the location of the camera.)

Any commands not stated on this page, if used, will be considered incorrect, and the- user will receive an “incorrect command” response.

ADMIN CONSOLE CONTINUED…

After a command is used, successfully or unsuccessfully, a pop-up window will appear stating that it has worked or not worked, and in some cases, will display the information you have requested. Such as the “GET RESO”, “SHOW FEED” and “GET IMAGE LOGS”.

After the “GET IMAGE LOGS” command has been used, a pop up will appear stating the full folder size and  that you are attempting to download. you will be given a download button. If download is pressed, close pop up and show a status bar indicating how many images have been downloaded. Once complete, close the status bar popup and open another saying “download successful” or “download unsuccessful”.

After the “GET FEED” command has been used, show a popup showing the live camera footage. Nothing else. The way you close is just like a normal window.

LOGGING

Logging means creating a file that keeps track of a target or set of data. In this camera/case, logging means keeping track of everything. Such as; connection attempts from an official or unofficial client, images taken and what was detected in those too. As well as timestamps and dates for everything. Images older than 14 days will be deleted with the consent of the root user.

BACKEND

The backend of this console will be built with the “websockets” PyPi package. The camera will always be listening for a host and port connection on [private_ip:293].

A router port will also need to be opened to direct to the camera's private ip address. However, this will be unnecessary if the camera and console/viewer/client is on the local network.

If an incorrect command is received, the server/camera should send a “Error Code 4:/ INVALID COMMAND”. command, meaning “incorrect command”. To avoid LUDICROUS amounts of data being used with live streaming, I plan to encrypt the live stream to a numpy array, to be decrypted as an image in the admin console. Every image will be sent/received exactly like this.

ENCRYPTION

The encryption standard will be a 4096 bit public and private key pair for the client and server. “https” is planned to be used but if need be, we will use “http”. Nevertheless, it will stay as a 4096 bit key with public and private key decryption. The login credentials will be hashed and stored on the raspberry pi’s local storage. For example; “user1:pass1” becomes “(asked thingy here.)”or whatever. The “user1” is the username of course and the same goes for the password. The should be entered into the admin console as “(username):(password)”.
